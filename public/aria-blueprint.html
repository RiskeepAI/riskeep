Configuración de API Key en Binance

Permisos requeridos, qué bloquear y cómo evitar baneos por Rate Limits

✅

ACTIVAR

  • ✓

  • ✓
    Enable Futures — Abrir y cerrar posiciones en Binance Futures
  • ✓
    IP Restriction — Whitelist de la IP de tu máquina (imprescindible)
  • ✓

  • ✓

🚫

NUNCA ACTIVAR

  • ✗
    Enable Withdrawals — El bot NUNCA necesita retirar fondos
  • ✗
    Internal Transfers — Innecesario y peligroso si la clave se filtra
  • ✗
    Sin IP whitelist — Una clave sin restricción de IP es accesible desde cualquier lugar
  • ✗
    Misma key testnet/mainnet — Entornos separados, siempre
  • ✗
    Keys en código / git — Solo en ~/.env, fuera del repositorio
⚡

Rate Limits — Cómo evitar el baneo de IP

  • →
    WebSocket sobre REST: streams de velas sin límite de peso frente a 1200 pts/min en REST.
  • →
    Header X-MBX-USED-WEIGHT: monitorízalo en cada respuesta para saber el consumo real.
  • →
    Stream combinado: /stream?streams=btcusdt@kline_5m/ethusdt@kline_5m — una sola conexión para múltiples pares.
  • !
    Error 429 = límite alcanzado. Espera mínima de 1 minuto antes de reintentar.
  • !
    Error 418 = IP baneada. Binance bannea automáticamente si sigues enviando peticiones tras un 429.
  • →
    Exponential backoff: el launcher implementa reintentos con espera exponencial para evitar acumular errores.
🔒

Principio de mínimo privilegio

La API Key del bot solo puede operar en Futures. No puede retirar, transferir ni acceder a otras cuentas. Ante una clave comprometida, el peor escenario posible es pérdida de posiciones abiertas — nunca vaciado de cuenta.

Rotación recomendada: cada 90 días en producción. Revoca inmediatamente si detectas actividad inusual.

La Regla de la Bolsa Aislada

Cómo proteger tu capital total mientras ARIA opera de forma autónoma

03 / 05
🧱

Capital segregado

ARIA opera solo con la bolsa asignada. Si el sistema falla completamente, tu pérdida máxima es el tamaño de esa bolsa — el resto queda intacto.

📉

Drawdown máximo

Si la bolsa cae el % configurado, ARIA entra en pausa automática. No reanuda hasta que un humano revisa el estado y confirma la continuación.

🎯

Sizing adaptativo

Tras 3 pérdidas consecutivas, el tamaño de posición baja a ×0.5 automáticamente. Se recupera al tamaño normal tras 2 operaciones ganadoras.

🔄

Consolidación diaria

Las ganancias netas se consolidan en el balance total al final del día. El bot no compound-interea sobre ganancias no revisadas.

⚠️

Los 30 días en Testnet no son opcionales

Es el tiempo mínimo para que el LLM genere suficientes episodios de memoria y los LoRA adapters se especialicen en tu perfil de riesgo específico. Saltar este paso en modo REAL equivale a operar con un sistema sin calibrar.

🎮 Modos de operación
OBSERVE

Solo análisis — ARIA analiza el mercado y registra episodios pero no ejecuta ninguna orden. Ideal para verificar que el sistema funciona correctamente tras la instalación.

MANUAL

Semi-automático — ARIA propone la operación (símbolo, dirección, tamaño, stop, TP) y espera tu aprobación en el dashboard. Útil para aprender el funcionamiento del sistema antes de pasar a AUTO.

AUTO

Completamente autónomo — ARIA ejecuta dentro de los límites del Risk Manager sin intervención humana. El Risk Manager sigue actuando como barrera — ninguna orden pasa si viola los límites configurados.

PAPER

Modo AUTO en Binance Testnet — Sin dinero real. Las operaciones se ejecutan en testnet.binance.vision con créditos de prueba. Obligatorio 30 días antes de pasar a REAL.

REAL

Modo AUTO en Binance Mainnet — Capital real. Requiere mínimo 30 días en modo PAPER y configuración explícita de claves de mainnet. El sistema bloquea el cambio si no se cumplen los requisitos.

System Architecture

Complete flow from Binance data to order execution

01 / 05
Layer 1 · Data

Persistent WebSocket. Automatic reconnection. Real-time OHLCV without REST polling.

Layer 2 · Analysis

Score 0-100 that filters weak signals before invoking the LLM, saving unnecessary inference.

Layer 3 · Local AI

Fully offline LLM. LoRA adapters specialized in BTC patterns and risk management.

Layer 4 · Risk

Independent barrier from the LLM. If limits are exceeded, the order never reaches the market.

Binance API Key Setup

Required permissions, what to block and how to avoid Rate Limit bans

02 / 05
✅

ENABLE

  • ✓
    Enable Reading — Price, balance and open positions
  • ✓
    Enable Futures — Open and close positions on Binance Futures
  • ✓
    IP Restriction — Whitelist your machine's IP (mandatory)
  • ✓
    Separate Testnet Key — Create a dedicated API Key for testnet.binance.vision
  • ✓
    Enable Spot Trading — Only if you also trade spot
🚫

NEVER ENABLE

  • ✗
    Enable Withdrawals — The bot NEVER needs to withdraw funds
  • ✗
    Internal Transfers — Unnecessary and dangerous if the key is leaked
  • ✗
    No IP whitelist — A key without IP restriction is accessible from anywhere
  • ✗
    Same key testnet/mainnet — Separate environments, always
  • ✗
    Keys in code / git — Only in ~/.env, outside the repository
⚡

Rate Limits — How to avoid IP bans

  • →
    WebSocket over REST: candlestick streams with no weight limit vs 1200 pts/min on REST.
  • →
    Header X-MBX-USED-WEIGHT: monitor it on every response to know the real consumption.
  • →
    Combined stream: /stream?streams=btcusdt@kline_5m/ethusdt@kline_5m — a single connection for multiple pairs.
  • !
    Error 429 = limit reached. Minimum wait of 1 minute before retrying.
  • !
    Error 418 = IP banned. Binance automatically bans if you keep sending requests after a 429.
  • →
    Exponential backoff: the launcher implements retries with exponential wait to avoid accumulating errors.
🔒

Principle of Least Privilege

The bot's API Key can only operate in Futures. It cannot withdraw, transfer or access other accounts. In the event of a compromised key, the worst possible scenario is loss of open positions — never an emptied account.

Recommended rotation: every 90 days in production. Revoke immediately if you detect unusual activity.

The Isolated Wallet Rule

How to protect your total capital while ARIA operates autonomously

03 / 05
🧱

Segregated capital

ARIA only operates with the assigned wallet. If the system fails completely, your maximum loss is the size of that wallet — the rest remains untouched.

📉

Maximum drawdown

If the wallet falls the configured %, ARIA enters automatic pause. It does not resume until a human reviews the state and confirms continuation.

🎯

Adaptive sizing

After 3 consecutive losses, position size drops to ×0.5 automatically. It recovers to normal size after 2 winning trades.

🔄

Daily consolidation

Net profits are consolidated into the total balance at the end of the day. The bot does not compound on unreviewed gains.

⚠️

The 30 days on Testnet are not optional

This is the minimum time for the LLM to generate enough memory episodes and for the LoRA adapters to specialize in your specific risk profile. Skipping this step in REAL mode is equivalent to operating with an uncalibrated system.

🎮 Operating modes
OBSERVE

Analysis only — ARIA analyses the market and records episodes but executes no orders. Ideal to verify the system works correctly after installation.

MANUAL

Semi-automatic — ARIA proposes the trade (symbol, direction, size, stop, TP) and waits for your approval in the dashboard. Useful for learning how the system works before moving to AUTO.

AUTO

Fully autonomous — ARIA executes within Risk Manager limits without human intervention. The Risk Manager still acts as a barrier — no order passes if it violates the configured limits.

PAPER

AUTO mode on Binance Testnet — No real money. Trades are executed on testnet.binance.vision with test credits. Mandatory 30 days before switching to REAL.

REAL

AUTO mode on Binance Mainnet — Real capital. Requires a minimum of 30 days in PAPER mode and explicit mainnet key configuration. The system blocks the switch if requirements are not met.
